# Migrating from Lovable Cloud to Your Own Supabase

## Overview
This guide walks you through moving your app's backend from Lovable Cloud to a self-managed Supabase project.

## Prerequisites
- A Supabase account (free tier is fine)
- Your Vercel project ready for deployment

---

## 1. Create a Supabase Project

1. Go to [https://supabase.com](https://supabase.com) and sign in.
2. Click **New project**.
3. Pick an organization, name your project, choose a region, and set a database password.
4. Wait for provisioning (1–2 minutes).

---

## 2. Collect Credentials

In your new project's dashboard:

| Variable | Location |
|---|---|
| `SUPABASE_URL` | Settings → API → Project URL |
| `SUPABASE_PUBLISHABLE_KEY` | Settings → API → `anon` `public` key |
| `SUPABASE_SERVICE_ROLE_KEY` | Settings → API → `service_role` `secret` key |

> ⚠️ **Keep the service role key secret.** It bypasses RLS.

---

## 3. Run the Database Schema

1. Open the **SQL Editor** in your Supabase dashboard.
2. Click **New query**.
3. Paste the entire contents of `schema.sql` (from this repo).
4. Click **Run**.

This creates all tables, enums, indexes, RLS policies, triggers, and seed data.

### What gets created
- `app_role`, `material_status`, `transaction_status` enums
- `user_roles`, `profiles`, `profile_contacts`, `material_categories`, `materials`, `transactions`, `transaction_events`, `messages`, `notifications` tables
- Row Level Security (RLS) policies for every table
- Auto-profile creation on signup (`handle_new_user` trigger)
- Inventory adjustment trigger on transactions
- Notification triggers for new transactions, status changes, and messages
- A default admin user: `ab@bossman.construction` / `Testing@123`

---

## 4. Create the Storage Bucket

1. In Supabase dashboard, go to **Storage**.
2. Click **New bucket**.
3. Name it exactly: `material-photos`
4. Keep **Public** toggled **OFF**.
5. Click **Save**.

The SQL already added RLS policies for this bucket.

---

## 5. Configure Auth Providers (Optional)

### Email / Password
Enabled by default. No action needed.

### Google OAuth
1. Go to **Authentication → Providers → Google**.
2. Toggle **Enabled**.
3. Enter your Google Client ID and Secret.
4. Under **Authentication → URL Configuration**, add your redirect URLs:
   - `http://localhost:8080` (for local dev)
   - `https://your-vercel-app.vercel.app` (for production)

---

## 6. Update Environment Variables

### Local Development
Create `.env.local` in project root:

```bash
VITE_SUPABASE_URL=https://your-project.supabase.co
VITE_SUPABASE_PUBLISHABLE_KEY=your-anon-key

SUPABASE_URL=https://your-project.supabase.co
SUPABASE_PUBLISHABLE_KEY=your-anon-key
SUPABASE_SERVICE_ROLE_KEY=your-service-role-key
```

### Vercel Deployment
Add these in your Vercel project settings under **Environment Variables**:

| Name | Value |
|---|---|
| `VITE_SUPABASE_URL` | `https://your-project.supabase.co` |
| `VITE_SUPABASE_PUBLISHABLE_KEY` | `your-anon-key` |
| `SUPABASE_URL` | `https://your-project.supabase.co` |
| `SUPABASE_PUBLISHABLE_KEY` | `your-anon-key` |
| `SUPABASE_SERVICE_ROLE_KEY` | `your-service-role-key` |

Apply to **Production**, **Preview**, and **Development**.

> The auto-generated Supabase integration files (`src/integrations/supabase/*.ts`) already read from these variables. **No code changes are required.**

---

## 7. Verify Locally

```bash
bun run dev
```

Open `http://localhost:8080` and test:
- Sign up with email/password
- Check that a profile was auto-created
- Check that `user_roles` has the `recipient` role assigned

---

## 8. Deploy to Vercel

Push your code to GitHub (if not already), then:
1. Import the repo in Vercel.
2. Make sure the environment variables from Step 6 are set.
3. Build command: `bun run build`
4. Output directory: `.output`
5. Deploy.

---

## Troubleshooting

| Issue | Fix |
|---|---|
| "Missing Supabase environment variable" | Double-check all 5 env vars are set in Vercel / `.env.local` |
| 404 on Vercel | Make sure `vercel.json` has `"framework": "tanstack-start"` and output dir is `.output` |
| RLS permission denied | Run `schema.sql` fully; ensure GRANT statements executed |
| Auth not working | Verify `VITE_SUPABASE_PUBLISHABLE_KEY` matches the `anon` key from your project |
| Realtime not working | Confirm `supabase_realtime` publication includes `messages`, `transactions`, `notifications` |

---

## Note on Auto-Generated Files

The following files are auto-generated by Lovable and **should not be edited manually**:
- `src/integrations/supabase/client.ts`
- `src/integrations/supabase/client.server.ts`
- `src/integrations/supabase/auth-middleware.ts`
- `src/integrations/supabase/auth-attacher.ts`
- `src/integrations/supabase/types.ts`

These files dynamically read from your environment variables, so changing the env vars is all you need to switch Supabase projects.
